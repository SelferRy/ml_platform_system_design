==== Принципы IaC

Инфраструктура Enterprise ML Platform управляется через два уровня IaC с чётким разделением ответственности:

[cols="2,2,3,2"]
|===
| Уровень | Инструмент | Что управляет | Принцип

| Облачная инфраструктура
| Terraform
| VPC, EKS кластер, RDS, ElastiCache, MSK, S3 бакеты, IAM роли, security groups
| "Не существует без кода" — никакие ресурсы не создаются вручную через AWS Console

| K8s приложения
| Helm
| Deployments, Services, ConfigMaps, HPA, KEDA ScaledObjects, NetworkPolicies, ServiceAccounts
| "Не существует без chart" — все Kubernetes-объекты декларативны
|===

Принцип разделения: Terraform не знает о K8s-объектах выше уровня node group. Helm не знает о AWS-ресурсах ниже уровня IAM ServiceAccount (через IRSA). Граница — EKS endpoint + OIDC provider, которые Terraform передаёт в Helm через outputs.

Все изменения проходят через CI/CD пайплайн (см. раздел 4.02). Прямой доступ на production AWS аккаунт ограничен политиками IAM.

===== ML-специфика: GPU Node Group

GPU node group для Triton Inference Server:

[source,hcl]
----
resource "aws_eks_node_group" "gpu" {
  cluster_name    = aws_eks_cluster.main.name
  node_group_name = "gpu-workers"
  node_role_arn   = aws_iam_role.node.arn
  subnet_ids      = module.networking.private_subnet_ids

  instance_types = ["g4dn.xlarge"]   # 1× NVIDIA T4, 16 GB GPU, 4 vCPU, 16 GB RAM

  scaling_config {
    min_size     = 1
    max_size     = 5
    desired_size = 1
  }

  taint {
    key    = "nvidia.com/gpu"
    value  = "true"
    effect = "NO_SCHEDULE"
  }

  labels = {
    "node-type"       = "gpu"
    "workload-profile" = "inference"
  }
}
----

После создания node group устанавливаются:

* **NVIDIA device plugin DaemonSet** — экспонирует `nvidia.com/gpu` resource в K8s
* **Cluster Autoscaler** — настроен на GPU node group, scale-down delay 10 мин (холодный старт GPU)
* **DCGM Exporter DaemonSet** — GPU метрики для VictoriaMetrics (`mlp_gpu_utilization_percent`)

==== Bootstrap-последовательность

===== Порядок применения

----
Шаг 1: terraform apply -target=module.networking
        ↓ (создаёт VPC, subnets, NAT GW)

Шаг 2: terraform apply -target=module.eks
        ↓ (создаёт кластер, node groups, OIDC provider)

Шаг 3 (параллельно):
        terraform apply -target=module.rds
        terraform apply -target=module.elasticache
        terraform apply -target=module.msk
        terraform apply -target=module.s3
        terraform apply -target=module.iam
        ↓ (независимые ресурсы, зависят только от networking + iam)

Шаг 4: terraform apply -target=module.vault
        ↓ (Vault требует работающий EKS и RDS для dynamic credentials)

Шаг 5: helm install cert-manager (system/)
        ↓ (TLS сертификаты нужны всем сервисам)

Шаг 6 (параллельно):
        helm install vault-agent-injector
        helm install observability-stack (Grafana, VictoriaMetrics, Loki, Alertmanager)
        helm install keda
        helm install nvidia-device-plugin
        ↓ (системный слой)

Шаг 7: helm install platform/ (api-gateway, model-service, inference-service, ...)
        ↓ (платформенные сервисы)
----

===== Почему Vault до приложений

Vault устанавливается на Шаге 4, до любых платформенных сервисов, по двум причинам:

1. **Dynamic credentials**: при старте каждый pod запрашивает у Vault динамические credentials для PostgreSQL (TTL 1h). Если Vault недоступен — pod не может стартовать. Vault должен быть Ready до первого `helm install` платформы.
2. **PKI и TLS**: cert-manager использует Vault PKI backend для выпуска сертификатов. Vault PKI должен быть инициализирован до cert-manager (Шаг 5).

===== ML-специфика: Triton Inference Server

Triton требует особой конфигурации в Helm chart inference-service:

**Init-контейнер для загрузки модели из S3:**

[source,yaml]
----
initContainers:
  - name: model-loader
    image: amazon/aws-cli:2.15
    command:
      - sh
      - -c
      - |
        aws s3 sync s3://mlp-models-${ENV}/artifacts/${MODEL_ID}/${VERSION}/ \
          /model-repository/${MODEL_ID}/
    env:
      - name: MODEL_ID
        valueFrom:
          fieldRef:
            fieldPath: metadata.annotations['mlp/model-id']
    volumeMounts:
      - name: model-repository
        mountPath: /model-repository
----

Модели не упаковываются внутрь Docker образа — они загружаются в shared volume при старте пода. Это позволяет обновлять модели без пересборки образа.

**Startup probe с увеличенным таймаутом:**

[source,yaml]
----
startupProbe:
  httpGet:
    path: /v2/health/ready
    port: 8000
  failureThreshold: 60    # 60 × 10s = 600s максимум на загрузку модели
  periodSeconds: 10
  initialDelaySeconds: 30
----

**Пример Helm values для inference-service (prod):**

----

**GPU в dev: Triton CPU-only режим.** В dev окружении GPU node group отсутствует. Triton запускается с флагом `--backend-config=tensorflow,version=2` в CPU-only конфигурации. Это позволяет тестировать pipeline развёртывания без GPU ресурсов, но с деградацией производительности (~10× медленнее). Для тестов производительности используется staging.

==== Контейнеризация

===== Стратегия образов

[cols="3,2"]
|===
| Принцип | Реализация

| **Base images**
| Python сервисы: `python:3.11-slim`. Go сервисы: `golang:1.22-alpine` builder + `gcr.io/distroless/static` runtime. Обновление base images еженедельно через Dependabot.

| **Multi-stage build**
| Stage 1 (builder): установка зависимостей, компиляция. Stage 2 (runtime): только артефакты, без build tools. Уменьшает размер образа и attack surface.

| **Версионирование**
| Tag = commit SHA (8 символов). `latest` не используется в prod. Пример: `mlp/inference-service:a3f9b12c`. Дополнительно тег окружения: `mlp/inference-service:staging` для canary promotion.

| **ECR lifecycle**
| Хранить последние 10 образов per сервис. Образы старше 30 дней с тегом `staging` — удалять. Образы `prod-*` — хранить 1 год (аудит).
|===

==== Обновление ПО без простоев

===== Rolling Update

Стандартная стратегия для всех платформенных сервисов:

[source,yaml]
----
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxUnavailable: 0    # никогда не уменьшать ниже желаемого числа реплик
    maxSurge: 1          # одновременно поднять один дополнительный pod
----

Kubernetes поднимает новый pod, ждёт readiness probe, затем завершает один старый pod — и так по кругу. При `maxUnavailable: 0` в любой момент доступны все `replicaCount` реплик.

===== Сценарий для Triton (детализация)

Triton требует особого внимания из-за большого startup probe и долгого завершения:

----
1. Создаётся новый Triton pod
2. init-контейнер (model-loader) скачивает модель из S3 (30-120с)
3. Triton стартует, загружает модель в VRAM
4. startupProbe: GET /v2/health/ready, failureThreshold=60, period=10s
   → максимум 600с на готовность
5. Pod переходит в Ready → получает трафик через Service
6. Kubernetes посылает SIGTERM старому pod
7. PreStop hook: sleep 15s (drain in-flight запросы)
8. Triton завершает обработку текущих батчей
9. Контейнер завершается (terminationGracePeriodSeconds: 60)
----

===== Когда Rolling Update НЕ гарантирует zero-downtime

[cols="2,3,3"]
|===
| Сценарий | Проблема | Митигация

| **Схема БД**
| Новая версия ожидает колонку, которой нет. Или старая версия несовместима с новой схемой.
| Backwards-compatible migrations: только `ADD COLUMN DEFAULT NULL`. `DROP COLUMN` — в отдельном деплое через 2 недели.

| **API-контракт**
| Клиент отправляет запрос старого формата к новому pod или наоборот.
| API versioning `/v1/`, `/v2/`. Старый endpoint держится минимум 1 мажорный релиз.

| **Kafka schema**
| Новый consumer не может десериализовать сообщения старого формата.
| Schema Registry + backward-compatible эволюция (только `OPTIONAL` поля). `buf breaking` check в CI.

| **Модель Triton**
| Новая версия модели ожидает другой input shape или тип.
| Model версионирование в репозитории. Одновременно активны v1 и v2 — переключение через feature flag, откат через FSM (см. doc/3.01).
|===

===== Blue-Green как альтернатива

Для критических обновлений (смена major версии, breaking change в схеме БД, замена модели) используется Blue-Green:

1. Поднять полный второй stack (Green) параллельно с текущим (Blue)
2. Запустить smoke + integration тесты на Green
3. Переключить Load Balancer target group: 100% трафик на Green
4. Blue держать 15 мин как fallback
5. Удалить Blue stack

Blue-Green увеличивает затраты на ~2× в момент переключения. Для Baseline используется редко — только при breaking changes.

NOTE: GitOps (ArgoCD) и Canary (Flagger) рассматриваются как extension points для MVP фазы, когда появятся требования к progressive delivery.
