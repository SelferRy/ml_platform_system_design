== Слайд 2: Проблема и цель

=== Контекст

Крупные компании развёртывают десятки ML-моделей в production. Без единой платформы каждая команда решает одни и те же проблемы: версионирование, безопасное развёртывание, мониторинг качества, изоляция ресурсов.

=== Ключевые вызовы

* *Governance* — контроль жизненного цикла модели от Draft до Deprecated
* *Scale* — 100+ моделей, 50+ тенантов, 10 000 RPS пиковой нагрузки
* *Safety* — недопустимо развернуть некачественную модель в production без согласований
* *Multi-tenancy* — изоляция данных и ресурсов между командами/проектами
* *Observability* — детекция дрифта модели, SLO для каждого инференс-эндпоинта

=== Цель

Единая корпоративная платформа, которая обеспечивает:

. Управляемый жизненный цикл ML-моделей с workflow согласований
. Высокопроизводительный инференс (sync / async / streaming / batch)
. Мультитенантность с изоляцией данных и resource quotas
. Полный стек observability и безопасности класса Enterprise

=== Архитектурный стиль

*Микросервисы + Event-Driven* — ключевые обоснования:

* Inference Service масштабируется независимо от Model Service
* Сбой Deployment Service не влияет на работающий инференс
* Kafka обеспечивает развязку компонентов и обработку пиков нагрузки

== Слайд 5: Архитектура решения

image::ha-deployment.png[HA Deployment Diagram, width=750]

=== Ключевые паттерны

[cols="2,3"]
|===
| Паттерн | Применение

| State Machine
| Жизненный цикл модели: Draft → Validating → Staging → Production → Deprecated

| CQRS
| Redis (inference hot path, hit rate >80%) + PG read replica (UI/reporting)

| Saga (Orchestration)
| Транзакционный контроль квот через Temporal; компенсация при сбоях

| Circuit Breaker
| Inference Service → GPU Workers; предотвращение каскадных отказов

| Canary / Blue-Green
| Безопасное развёртывание: постепенный сдвиг трафика + мгновенный откат

| CDC (Debezium)
| PostgreSQL WAL → Kafka: decoupling Model Service и Deployment Service

| Scale-to-zero
| KEDA по Kafka lag: неактивные модели (0 RPS за 30 мин) → 0 реплик

| Rate Limiting
| Per-tenant лимиты; burst 2× на первые 30s; DLQ для rejected requests
|===

=== HA-топология

3 Availability Zones · Active-Active · RTO <2 мин при AZ failure · RPO=0 (sync PG replica)

== Слайд 7: Хранение данных

=== Стратегия хранения и CAP-позиция

[cols="2,2,2,3"]
|===
| Данные | Хранилище | CAP | Обоснование

| Метаданные моделей
| PostgreSQL (Multi-AZ)
| *CP*
| ACID-транзакции; sync replica RPO=0; sources of truth

| Артефакты моделей
| S3 / MinIO
| *AP*
| Strong consistency для новых PUT; 11 девяток durability; lifecycle to Glacier

| Кэш предсказаний
| Redis (Sentinel)
| *AP*
| Sub-ms latency; TTL; не источник истины — восстанавливается из PG

| События / Аудит
| Kafka → PostgreSQL
| *AP*
| Decoupling; replay; CDC-ready; retention 7 дней

| Метрики
| VictoriaMetrics
| *AP*
| 10:1 сжатие vs Prometheus; горизонтальное масштабирование; PromQL совместим

| Batch Data
| S3 (BYOB / Managed)
| *AP*
| Enterprise: cross-account IAM; self-service: managed buckets с tenant prefix
|===

=== CQRS на двух уровнях

* *Уровень 1 (hot path)*: Inference → Redis hit (>80%) → ответ без PG
* *Уровень 2 (model registry)*: Write → PG primary; Read UI/API → PG async replica (AZ-C)

== Слайд 10: Риски и компромиссы

=== Топ-5 рисков

[cols="1,3,1,1,3"]
|===
| ID | Риск | Вероят. | Влияние | Митигация

| R-01
| Нехватка GPU при всплесках
| Средняя
| Критическое
| Reserved capacity · CPU fallback · scale-to-zero для неактивных

| R-02
| Необнаруженный дрифт модели
| Высокая
| Значительное
| Drift Detector → алерт P2 · быстрый откат одним действием

| R-04
| Сбой развёртывания в production
| Средняя
| Критическое
| Canary + health check gate · auto-rollback при error >1%

| R-05
| Нарушение безопасности
| Низкая
| Критическое
| RBAC + RLS + Vault · tenant isolation tests в CI · AES-256

| R-15
| Noisy neighbor среди тенантов
| Средняя
| Значительное
| Per-tenant rate limits · K8s ResourceQuota · burst 2× только 30s
|===

=== Ключевые trade-off

[cols="2,3,3"]
|===
| Решение | Выигрыш | Компромисс

| Микросервисы vs монолит
| Независимое масштабирование, изоляция отказов
| Операционная сложность, сетевые накладные расходы

| PostgreSQL CP vs eventual
| RPO=0, ACID-транзакции, источник истины
| Вертикальный bottleneck при >50K RPS

| Redis кэш (AP)
| Sub-ms inference hit path, >80% cache hit
| Split-brain допустим — Redis не источник истины

| Шардирование PG отложено
| Нет избыточной сложности на Baseline
| Структурный рефакторинг при >50K RPS sustained

| Baseline: explicit Kafka publish (не CDC)
| Меньше операционной сложности (нет Debezium)
| При росте consumers — coupling растёт; CDC — MVP extension point
|===
