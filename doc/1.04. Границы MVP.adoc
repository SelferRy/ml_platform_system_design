
=== Границы MVP

==== Обоснование MVP-среза

Исходный список требований охватывает ~60 FR в 10 доменах — это продуктовое видение, а не MVP. При проектировании MVP применялись три принципа отбора:

1. **Блокирует ли передачу первому клиенту?** Если да — MVP. Если нет — Future.
2. **Несёт ли отсутствие неприемлемые операционные риски?** Если да — MVP (FR-207 откат, FR-303 health check).
3. **Можно ли закрыть вручную или процессом?** Если да — Future (self-service онбординг, cron, chargeback).

Мультирегиональность (FR-1001–1006), детекция дрифта (FR-704), Billing Saga (FR-604) и streaming (FR-103) выведены в Future: они не блокируют первого клиента и требуют непропорционально большой архитектурной проработки.

==== Фаза 2 — Future Releases

*Цель:* масштабирование до enterprise, мультирегиональность, продвинутый ML ops.

[cols="2,3"]
|===
| Требование | Причина откладывания

| Потоковый вывод для LLM (FR-103)
| Нужен только при добавлении LLM use-cases; требует WebSocket/SSE инфраструктуры

| A/B тестирование (FR-105), Canary с автооткатом (FR-106)
| После стабилизации мониторинга; opaque без метрик дрифта

| Graceful degradation (FR-107), Кэширование предсказаний (FR-108)
| Оптимизация; не блокирует первого клиента

| Отслеживание прогресса batch и частичные результаты (FR-403)
| Удобство; можно смотреть статус через статус задачи

| Checkpointing batch-задач (FR-407)
| Сложная реализация; полный перезапуск приемлем (NFR-208)

| Планирование задач/cron (FR-408)
| Можно использовать внешний scheduler

| Изоляция вычислений (FR-503), Self-service онбординг (FR-507)
| Enterprise tier; ручной онбординг приемлем для MVP

| Транзакционный контроль квот: Saga (FR-604)
| Учёт (FR-601/602) нужен в MVP; Saga — сложная распределённая транзакция только при автоматическом enforcement

| Бюджетные квоты с enforcement (FR-603), оповещения (FR-605), отчёты (FR-606)
| Ручной контроль через дашборд приемлем на старте

| Распределённая трассировка (FR-703)
| Нужна при сложных цепочках; достаточно request_id в логах

| Детекция дрифта (FR-704, FR-705), Quality monitoring (FR-706, FR-709)
| Не блокирует запуск; первый клиент мониторит качество самостоятельно

| OAuth 2.0, SSO (FR-801 в части SSO)
| Enterprise auth; API ключи достаточны на старте

| PII, data residency, compliance, scan (FR-805–FR-808)
| Регуляторные требования; FR-1005 — Backlog (один регион достаточен для MVP)

| Sandbox-среда (FR-909)
| Использовать dev-окружение вместо отдельной sandbox; см. обоснование ниже

| Вся глобальная инфраструктура (FR-1001–FR-1006)
| Один регион с backup покрывает 95% use-cases первого клиента

| CLI (FR-906), SDK (FR-907)
| REST API + документация достаточны на старте

| Multi-cloud (FR-1007)
| Одного облака достаточно для MVP
|===

NOTE: FR-909 (Sandbox-среда): в backlog MVP предполагалось использовать "staging-среду", однако staging — это окружение деплоя, а не sandbox для экспериментов клиента. На этапе Future будет реализована отдельная sandbox с изолированными квотами и mock-инференсом.

NOTE: FR-1005 (Enforcement резидентности данных): добавлен в doc/1.01 как отдельное требование домена "Глобальная инфраструктура" с приоритетом Could Have. Будет реализован при выходе на второй регион вместе с FR-1001–FR-1004.
